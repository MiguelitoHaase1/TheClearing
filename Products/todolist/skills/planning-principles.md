# Planning Principles — How Michael Plans His Days

These principles are extracted from the /morning skill. Apply them when helping Michael think about what to work on, when triaging tasks, or when he asks for planning help via iMessage or CLI.

---

## One Priority Per Day

Every day has ONE priority — not a list. The priority is a statement like "Ship the investor deck" or "Deep work on CRM data model." All tasks are evaluated against it.

When helping Michael decide what matters, reason through these lenses:
- **Deadlines** — What has the tightest deadline today or tomorrow?
- **Importance** — What has the highest impact regardless of urgency?
- **Momentum** — What has Michael been focused on recently? Is there value in continuing that thread?
- **Patterns** — What keeps getting deferred? What gets dropped?
- **Calendar pressure** — Do meetings constrain deep-work time?

## Task Scheduling Rules

- **Higher priority = earlier in the day.** P0/P1 tasks go in the 09:00–12:00 deep work window.
- **Duration estimates:** P0/P1 tasks get 60 min. P2/P3 tasks get 30 min.
- **Family time (17:00–19:30) is protected.** Never schedule tasks there.
- **Evening window (19:30–22:00) is for lighter work.** Personal tasks, learning, exercise.
- **Every task gets a slot or an explicit recommendation.** Never silently drop a task. If it doesn't fit, say "reschedule to [date]" or "consider dropping."

## Task Categories

Each task relates to the day's priority in one of three ways:
- **Direct** — Directly advances today's priority
- **Indirect** — Supports the priority or related goals
- **Obligation** — Unrelated to the priority (meetings, errands). Label honestly. Don't manufacture fake connections.

## Tunnel Vision Detection

If the same priority topic has been active for 3+ consecutive days, flag it. Not as a problem — as a pattern to notice. Ensure at least one alternative is visible.

## Overflow Handling

When there are more tasks than time:
1. Schedule highest-priority tasks first
2. For tasks that don't fit:
   - **High priority** → Reschedule to next workday
   - **Low priority, not time-sensitive** → Recommend dropping
3. Never silently drop. Every task gets a decision.

## Today vs. Deferred

- **Today's tasks** — Due on target date + overdue. These get scheduled.
- **Deferred tasks** — Future dates or no date. Visible but separate. Lightweight — no enrichment, no time assignments.

## Relationship Awareness

When tasks touch the same topic from different angles, they benefit from being:
- Linked (`td link`) so context carries across
- Done back-to-back (batching)
- Sequenced if one is a prerequisite for the other

Near-duplicates should be merged — keep the higher-priority version, mark the other done.

## Challenge Thinking

When a priority choice conflicts with reality — a deferred high-priority task, a looming deadline, a pattern of avoidance — surface it as a brief observation. One sentence. Reference specific tasks, not generic advice. Non-blocking: state it and move on.

## Guardrails

- **No scoring algorithms.** Reason about priorities, don't calculate them.
- **One priority only.** If torn, pick one and mention the runner-up.
- **Landscape over directives.** Show the full picture. Let Michael decide.
- **Graceful degradation.** If data is missing, work with what you have.
- **Feedback is opt-in.** Only when Michael asks. Never proactive.
